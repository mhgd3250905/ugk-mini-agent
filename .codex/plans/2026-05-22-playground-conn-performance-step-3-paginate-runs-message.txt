本步目标：
- Add Bounded Run History Pagination
- `/v1/conns/:connId/runs` 不要一次返回全部历史。
- 前端默认只请求第一页，例如 `limit=10`，并支持“加载更多”。

当前真实背景：
- `apiFetchRuns(connId)` 当前直接请求 `/v1/conns/:connId/runs`。
- `GET /v1/conns/:connId/runs` 当前调用 `listRunsForConn(connId)` 并返回完整历史。
- `renderRunHistory()` 只展示 `runs.slice(0, 10)`，所以现在是“下载一大锅，只盛一小勺”，很浪费。
- `/runs/:runId/events` 已经有 event pagination；不要把它误当成本步已经完成。

实现要求：
- 给 run-list route 增加可选 `limit` 和稳定 cursor 查询参数。
- 保持不传 query 参数的旧调用向后兼容。
- cursor 必须遵守现有排序：`scheduled_at DESC, created_at DESC, run_id DESC`；不要只用时间戳，时间并列会翻车。
- 返回 `hasMore`、`nextBefore` 或等价分页元数据，并补 `src/types/api.ts`。
- 前端 `apiFetchRuns(connId)` 默认请求有界第一页。
- run history 区域支持“加载更多”，追加数据，不重置 selected conn、expanded run 或滚动位置。
- 不改数据库 schema，优先复用现有排序和存储能力。
